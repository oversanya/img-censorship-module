# PLAN.md — план развития Censor Module

План улучшения текущего MVP. Пункты сгруппированы по приоритету. Внутри каждого:
**проблема → что сделать → как проверить**. Начинать сверху.

Связанный контекст: [ARCHITECTURE.md](ARCHITECTURE.md) (как всё устроено сейчас).

## P0 — Критично (без этого «модерация» не модерирует)

### P0.1 Подключить реальный текстовый гард вместо заглушки
- **Проблема.** `adapters/text_stub.py` всегда возвращает «safe». Промпт и
  OCR-текст не проверяются — главная функциональная дыра.
- **Что сделать.**
  - Завести `adapters/text_classifier.py` с реальной моделью (варианты:
    `unitary/toxic-bert`, мультиязычный `martin-ha/toxic-comment-model`, или
    LLM-судья). Маппить выходы модели в наши коды категорий через таксономию.
  - Оставить `TextGuardStub` как фолбэк, когда модель недоступна (как в других
    адаптерах: `skipped` вместо падения).
  - Подключить в `pipeline.py` вместо/рядом со стабом; OCR-текст гонять туда же.
- **Проверка.** Юнит-тест: токсичный промпт → категория поднимается;
  безобидный → пусто. E2E: запрос с матерным `prompt` уходит в review/block.

### P0.2 Калибровка zero-shot классификатора
- **Проблема.** CLIP zero-shot softmax-ит оценки по меткам (сумма ≈ 1),
  поэтому абсолютные пороги 0.55/0.85 к ним применять некорректно — у любой
  картинки «горит» какая-то категория.
- **Что сделать (любой из вариантов, по возрастанию усилий).**
  1. Добавить «нейтральные» якорные метки (`a normal safe photo`,
     `everyday neutral image`) в `candidate_labels`, чтобы безобидные картинки
     набирали вес на них, а не на категориях нарушения. В `SignalResult`
     отдавать только превышение над нейтральным якорем.
  2. Откалибровать пороги на небольшом размеченном наборе (см. P1.3) — отдельные
     пороги для visual-сенсора, отличные от explicit-детектора.
  3. Долгосрочно — заменить zero-shot на дообученный мульти-лейбл классификатор.
- **Проверка.** Прогон по набору безобидных картинок → доля ложных
  review/block заметно падает; по «грязным» → recall не проседает.

### P0.3 Сделать API асинхронным / не блокировать event loop
- **Проблема.** `POST /v1/moderate` синхронный, инференс блокирует FastAPI.
- **Что сделать.** Выполнять `pipeline.moderate` в пуле потоков
  (`await run_in_threadpool(...)` или `asyncio.to_thread`), либо вынести инференс
  в отдельный воркер/очередь. Добавить таймаут на сенсор.
- **Проверка.** Нагрузочный тест: `/health` остаётся отзывчивым во время
  тяжёлого `/v1/moderate`.

## P1 — Качество и наблюдаемость

### P1.1 Логирование, метрики, аудит решений
- **Проблема.** Сейчас невозможно разобрать инцидент пост-фактум.
- **Что сделать.** Структурное логирование (request_id, вердикт, категории,
  тайминги сенсоров). Метрики Prometheus (кол-во block/review/allow, латентность
  по сенсорам, доля skipped/error). Опционально — сохранять решения в БД/лог-стор.
- **Проверка.** На дашборде видно распределение вердиктов и p95 латентности.

### P1.2 Расширить тесты
- **Проблема.** Покрыт только `DecisionEngine`.
- **Что сделать.**
  - Юнит-тесты каждого адаптера с замоканной моделью (ok / skipped / error).
  - Тест `pipeline.moderate` целиком на фейковых сенсорах.
  - Тест API через `fastapi.testclient`.
  - Граничные случаи: пустой запрос, битый base64, картинка без текста.
- **Проверка.** `python -m pytest`, покрытие ключевых веток.

### P1.3 Завести оценочный датасет и метрики качества
- **Проблема.** Нет способа измерить, хорошо ли модерация работает.
- **Что сделать.** Собрать небольшой размеченный набор (safe / по категориям).
  Скрипт, считающий precision/recall/FPR по категориям и подбирающий пороги.
- **Проверка.** Воспроизводимый отчёт «качество vs пороги».

### P1.4 Дифференцировать политику по `stage` и `scenario`
- **Проблема.** input и output проверяются одинаково; сценарии не учитываются.
- **Что сделать.** Передавать `request` в `DecisionEngine` (уже передаётся) и
  ввести профили порогов: например, на `output` строже, на `img2img_editing`
  учитывать исходную картинку. Вынести профили в конфиг.
- **Проверка.** Тесты: один и тот же сигнал даёт разный вердикт на input/output
  согласно профилю.

## P2 — Развитие моделей

### P2.1 Подключить ShieldGemma (или аналог) как настоящего судью
- **Проблема.** `ShieldGemmaJudge` — заглушка, всегда `skipped`.
- **Что сделать.** Реализовать загрузку/инференс мультимодальной модели по
  паттерну ленивой загрузки (как в `explicit_detector.py`). Судья получает
  картинку + промпт + сводку сигналов и выдаёт категории. Фасад уже готов к
  переключению: как только вернётся `ok`, эвристика не вызывается.
- **Проверка.** При `CENSOR_ENABLE_POLICY_JUDGE=true` ответ содержит сигнал
  `policy_judge_shieldgemma` со `status=ok`.

### P2.2 Кэширование результатов по хешу картинки
- **Проблема.** Одинаковые картинки переинференсятся.
- **Что сделать.** Кэш (in-memory/Redis) по `sha256` изображения + версии моделей.
- **Проверка.** Повторный запрос той же картинки отвечает из кэша (по метрике/логу).

### P2.3 Гибкая загрузка/выгрузка моделей и управление памятью
- **Что сделать.** Контроль устройства (CPU/GPU), выбор `dtype`, прогрев моделей
  на старте (опционально), graceful-обработка OOM.

## P3 — Эксплуатация и harden

- **Валидация входа.** Лимит размера картинки/строки base64, белый список
  форматов, защита от «декомпресс-бомб» в Pillow.
- **Версионирование политики.** В ответ добавить версии моделей и порогов, чтобы
  решения были воспроизводимы.
- **Контейнеризация.** Dockerfile + образ с предзагруженными весами и Tesseract.
- **Rate limiting / authn** на API, если он смотрит наружу.

## Рекомендуемый порядок

1. **P0.1** (текстовый гард) и **P0.2** (калибровка) — без них модерация
   формальна.
2. **P0.3** (асинхронность) — до любой нагрузки.
3. **P1.2 + P1.3** (тесты + датасет) — чтобы дальнейшие изменения не ломали качество.
4. **P1.1** (наблюдаемость), затем **P1.4** (профили политики).
5. **P2.1** (ShieldGemma) — когда нужен скачок качества и есть ресурсы под модель.
6. **P2.2–P3** — по мере выхода в прод.
