# Как устроен Censor Module под капотом

Документ объясняет, что реально происходит внутри MVP, шаг за шагом. Цель —
чтобы после прочтения было понятно: какие компоненты есть, кто кого вызывает,
где принимается решение и где сейчас «дыры».

> TL;DR. Это многослойный фильтр для картинок. На вход приходит запрос
> (промпт и/или картинка), несколько независимых «сенсоров» оценивают её по
> списку запрещённых категорий, а финальный движок по порогам выдаёт вердикт
> **allow / review / block**.

---

## 1. Общая картина

```
            ┌─────────────────────────────────────────────────────────┐
   HTTP /   │                  GuardrailPipeline.moderate              │
   CLI      │                                                         │
   запрос ─►│  load_image(request)                                    │
            │      │                                                  │
            │      ▼                                                  │
            │  ┌── Сенсоры (каждый возвращает SignalResult) ──────┐   │
            │  │  text_guard      (промпт → лексикон ru/en + ML)  │   │
            │  │  ocr             (текст внутри картинки)         │   │
            │  │  text_guard      (OCR-текст → лексикон ru/en)    │   │
            │  │  visual          (zero-shot, КАЛИБРОВАН к safe)  │   │
            │  │  explicit        (NSFW-детектор)                 │   │
            │  │  policy_judge    (fusion + эскалация ShieldGemma)│   │
            │  └──────────────────────┬───────────────────────────┘   │
            │                         ▼                               │
            │                 DecisionEngine.decide                   │
            │                         │                               │
            │                         ▼                               │
            │      ModerationResponse: verdict + categories +         │
            │      confidence + evidence + signals + notes            │
            └─────────────────────────────────────────────────────────┘
```

Ключевая идея: **сенсоры ничего не решают сами**. Каждый лишь выдаёт оценки по
категориям в едином формате `SignalResult`. Всю «политику» (что блокировать, что
на ревью) держит один компонент — `DecisionEngine`. Это позволяет менять модели,
не трогая логику решений.

## 2. Формат данных — три pydantic-модели (`schemas.py`)

Весь модуль общается тремя структурами:

### `ModerationRequest` — вход
| поле | смысл |
|------|-------|
| `request_id` | автогенерируемый UUID запроса |
| `scenario` | `text2image` / `img2img_stylization` / `img2img_editing` / `output` |
| `stage` | `input` (проверка до генерации) или `output` (проверка результата) |
| `prompt` | текст промпта (опционально) |
| `image_path` / `image_base64` | картинка как путь к файлу или base64 |
| `metadata` | произвольные доп. данные |

### `SignalResult` — то, что возвращает каждый сенсор
| поле | смысл |
|------|-------|
| `name` | имя сенсора (`visual_classifier`, `ocr_adapter`, …) |
| `status` | `ok` (отработал) / `skipped` (выключен/нет бэкенда) / `error` (упал) |
| `categories` | словарь `{код_категории: оценка 0..1}` |
| `text` | извлечённый текст (для OCR) |
| `reason` | человекочитаемое пояснение |
| `raw` | сырой вывод модели для отладки |

> Только сигналы со `status == "ok"` участвуют в голосовании. `skipped`/`error`
> не валят запрос — их причина просто попадает в `notes` ответа. Это и есть
> «мягкая деградация»: нет модели → сервис всё равно отвечает.

### `ModerationResponse` — выход
`verdict` (allow/review/block), `categories` (что сработало), `confidence`
(макс. оценка по сработавшим категориям), `reason`, `evidence` (какие сенсоры
подтвердили каждую категорию), `signals` (все сырые сигналы), `notes`.

## 3. Таксономия нарушений (`taxonomy.py`)

11 категорий, у каждой есть флаг `hard_block`:

**Hard-block (блокируются от любого сенсора):**
`sexual`, `violence_gore`, `self_harm`, `hate_extremism`, `illegal_activity`,
`shocking`.

**Soft / review (блокируются только если подтвердил policy judge):**
`harassment`, `deception_fraud`, `political_persuasion`,
`health_misinformation`, `spam_scams`.

Из этого списка автоматически строятся вспомогательные структуры:
- `VISUAL_LABELS` — текстовые описания категорий, которые скармливаются
  zero-shot классификатору в качестве вариантов ответа;
- `VISUAL_LABEL_TO_CODE` — обратный маппинг «описание → код»;
- `HARD_BLOCK_CATEGORIES` / `SOFT_REVIEW_CATEGORIES` — множества для движка решений.

Добавить новую категорию = добавить один `CategorySpec`, всё остальное
подтянется автоматически.

## 4. Конфигурация (`config.py`)

`Settings` — это `frozen dataclass`, который читает переменные окружения **один
раз при импорте**. Все фичи можно включать/выключать без правки кода:

| переменная | по умолчанию | что делает |
|------------|--------------|------------|
| `CENSOR_ENABLE_OCR` | `true` | включить OCR |
| `CENSOR_ENABLE_VISUAL_CLASSIFIER` | `true` | включить zero-shot классификатор |
| `CENSOR_ENABLE_EXPLICIT_DETECTOR` | `true` | включить NSFW-детектор |
| `CENSOR_ENABLE_POLICY_JUDGE` | `false` | включить ShieldGemma (пока не подключён) |
| `CENSOR_BLOCK_THRESHOLD` | `0.85` | порог блокировки |
| `CENSOR_REVIEW_THRESHOLD` | `0.55` | порог ручной проверки |
| `CENSOR_VISUAL_MODEL_ID` | `openai/clip-vit-base-patch32` | модель классификатора |
| `CENSOR_EXPLICIT_MODEL_ID` | `Falconsai/nsfw_image_detection` | модель NSFW |
| `CENSOR_POLICY_JUDGE_MODEL_ID` | `google/shieldgemma-2-4b-it` | модель-судья |
| `CENSOR_HF_CACHE_DIR` | `<проект>/.cache/huggingface` | куда качать веса |
| `CENSOR_TESSERACT_CMD` | автопоиск | путь к бинарю Tesseract |

`_default_tesseract_cmd()` сам ищет Tesseract: сначала в `PATH`, потом в типовых
путях Windows/macOS/Linux.

> ⚠️ Так как настройки читаются на уровне определения класса при импорте,
> менять переменные окружения «на лету» в одном процессе бесполезно — нужно
> перезапускать. Это нормально для сервиса, но об этом стоит помнить в тестах.

## 5. Сенсоры (адаптеры) — `adapters/`

Все адаптеры устроены по одному паттерну: **ленивая загрузка модели + единый
возврат `SignalResult` + защита от падений**.

### 5.1 `text_classifier.py` — текстовый гард (`text_guard_heuristic`)
Реальная (а не заглушечная) проверка текста промпта **и** OCR-текста. Два слоя
сливаются по максимуму:
- **ML-слой (основной)**: HF text-classification модель, по умолчанию
  `cointegrated/rubert-tiny-toxicity` (~50МБ, быстрая на CPU, RU). Её метки
  (`insult/obscenity/threat/dangerous/non-toxic`) маппятся в таксономию по
  `ML_LABEL_MARKERS` (insult/obscenity → harassment, threat → violence_gore;
  `dangerous` намеренно не маппится как шумный, safe-метки пропускаются). Модель
  задаётся через `CENSOR_TEXT_MODEL_ID`; toxicity-модели читаются как
  мультилейбловые (`function_to_apply="sigmoid"`).
- **лексиконный слой (добивка)**: стемы запрещённых тем из `taxonomy.TEXT_LEXICONS`
  (ru + en). Нужен прежде всего там, где toxicity-модель не называет категорию —
  например «sexual» (запрос на порнографию модель видит как insult, а лексикон
  ставит именно `sexual`).

Нет `transformers`/сети/модели → тихо деградируем к одному лексикону (сервис не
падает). Если `CENSOR_TEXT_MODEL_ID` пуст — работает только лексикон.

### 5.2 `ocr.py` — извлечение текста из картинки
Через `pytesseract` (обёртка над движком Tesseract) вытаскивает строки текста.
Если `pytesseract` не установлен или Tesseract не найден — `skipped`. Сам по себе
OCR ничего не модерирует; найденный текст уходит в текстовый гард
(`ocr_text_guard_heuristic`), который теперь реально его проверяет.

### 5.3 `visual_classifier.py` — zero-shot классификатор (КАЛИБРОВАН)
По умолчанию `CLIP`. Модель не обучалась на наших категориях: мы передаём ей
описания всех 11 категорий **плюс несколько нейтральных safe-якорей**.

Сырой zero-shot softmax-ит оценки по всем меткам (сумма ≈ 1.0), поэтому у любой
картинки всегда есть «самая вероятная» категория нарушения — сравнивать такие
числа с порогом нельзя. Поэтому адаптер **не отдаёт сырой softmax**: он калибрует
оценки относительно safe-якоря (`calibration.calibrate_against_safe`):

```
p = s_c / (s_c + s_safe);   calibrated = clamp((p − floor) / (1 − floor), 0, 1)
```

Безобидная картинка (категория едва обгоняет safe) → ~0; картинка, где категория
уверенно доминирует над safe → ~1. Теперь оценку можно сравнивать с порогами и
сливать с другими сенсорами на единой шкале. Сырой softmax лежит в `raw["softmax"]`.

### 5.4 `explicit_detector.py` — NSFW-детектор
По умолчанию `Falconsai/nsfw_image_detection`. В отличие от zero-shot — это
нормальная обученная модель `image-classification` с откалиброванными
вероятностями по своим меткам (`normal` / `nsfw`). Берём максимум среди меток,
содержащих маркеры `nsfw/porn/hentai/sexy/explicit`, и кладём в категорию
`sexual`. Это самый надёжный сенсор в текущем наборе.

### 5.5 `policy_judge.py` + `fusion.py` — арбитр (fusion + эскалация)
`PolicyJudge` делает две вещи:

1. **Сведение (`fusion.fuse`)** — взвешенный noisy-OR по откалиброванным оценкам
   сенсоров. Для категории c с оценками p_i и весами надёжности w_i:

   ```
   score(c) = 1 − Π_i (1 − w_i · p_i)
   ```

   Это заменило прежнюю эвристику с магическими `±` и бессмысленным
   `evidence_count` (который из-за softmax-шума CLIP был константой при любой
   картинке). Теперь: согласие сенсоров естественно повышает уверенность; вклад
   каждого сенсора виден в `raw["contributions"]`; «согласие» (`agreement`)
   считается только по сигналам выше `EVIDENCE_FLOOR`, поэтому шум за улику не
   выдаётся. Итог пакуется в сигнал `policy_fusion`.

2. **Эскалация на ShieldGemma** — только на пограничных случаях
   (`escalation_needed`: «серая зона» между review и block, либо стадия `output`
   с любым риском). ShieldGemma — это policy-aware VLM, доп-слой судьи, а **не**
   аналог CLIP. Если он ответил (`ok`) — fusion пере-сводится уже с его участием
   (вес 1.0). Сейчас инференс не подключён → честный `skipped` (без подмены
   результата фейковым фолбэком). Включается флагом `CENSOR_ENABLE_POLICY_JUDGE`.

## 6. Движок решений (`decision.py`)

Финальный и самый важный шаг. Алгоритм:

1. **Источник оценок — сигнал `policy_fusion`** (откалиброванный noisy-OR). Берём
   его `categories` как оценки, `raw["agreement"]` как число согласных сенсоров и
   `raw["contributions"]` как источники. Если fusion-сигнала нет (движок вызвали
   изолированно в тесте) — фолбэк на максимум по сенсорам.
2. **Hard-блок.** Любая hard-категория со сведённой оценкой `≥ block_threshold` → блок.
3. **Soft-блок.** Soft-категория блокируется только если оценка `≥ block_threshold`
   **и** есть подтверждение: согласие `≥ 2` независимых сенсоров **или** вердикт
   ShieldGemma. (Так одиночный шумный сенсор не заблокирует «мягкую» категорию.)
4. **Review.** Всё, что `≥ review_threshold`, но не дотянуло до блока → ручная проверка.
5. **Иначе** → `allow`.

`confidence` = максимальная оценка среди сработавших категорий (для `allow` —
максимум вообще, как «насколько близко подошли к границе»). `evidence` показывает,
какие именно сенсоры подтвердили каждую сработавшую категорию.

> Обратите внимание: **`scenario` и `stage` сейчас на решение не влияют** — они
> просто прокидываются в ответ. То есть проверка input и output по сути
> одинаковая. Это место для будущей дифференциации политик.

## 7. Точки входа

### HTTP API (`app.py`)
FastAPI с двумя ручками:
- `GET /health` → `{"status": "ok"}`;
- `POST /v1/moderate` → принимает `ModerationRequest`, возвращает `ModerationResponse`.

`pipeline = GuardrailPipeline()` создаётся **один раз** при старте — модели
шарятся между запросами.

> ⚠️ Ручка `moderate` **синхронная**, а инференс моделей — тяжёлый и
> блокирующий. Под нагрузкой это заблокирует event loop FastAPI. См. PLAN.md.

### CLI (`cli.py`)
`python -m censor_guard.cli --scenario output --image-path path/to.png`
Создаёт `GuardrailPipeline()` **на каждый вызов** и печатает JSON-ответ. Удобно
для локальной проверки одной картинки.

## 8. Зависимости

- `requirements.txt` — лёгкое ядро: FastAPI, uvicorn, pydantic, Pillow.
  Сервис **поднимется и без ML** — все ML-сенсоры просто будут `skipped`.
- `requirements-ml.txt` — тяжёлое: `transformers`, `torch`, `accelerate`,
  `pytesseract`. Нужно для реальной работы сенсоров. Tesseract-бинарь ставится
  отдельно (это не pip-пакет).

Такое разделение позволяет гонять каркас/тесты быстро, а ML подключать по мере
надобности.

## 11. Главные ограничения текущего MVP (честный список)

1. **ShieldGemma не подключён** — точка эскалации работает, но инференс не вшит,
   поэтому на пограничных случаях судья честно `skipped`, и решение держится на
   сведении сенсоров. Это главный оставшийся пробел.
2. **Текст-гард покрывает не все категории** — ML-модель (rubert-tiny-toxicity)
   сильна в harassment/threat, но не различает sexual/политику/мед-дезинфо;
   sexual держится на лексиконе (обходится перефразировкой). Для полного покрытия
   нужна моделька уровня OpenAI-moderation / LLM-guard (см. PLAN.md).
3. **`scenario` не влияет на политику; `stage` влияет лишь на эскалацию** — единых
   дифференцированных политик по сценариям пока нет.
4. **API синхронный** — тяжёлый инференс блокирует event loop.
5. **Нет логирования/метрик/аудита** — тяжело разбирать инциденты пост-фактум.
6. **Калибровка CLIP — линейная относительно одного safe-якоря** — работает, но
   порог `floor` стоит подбирать на реальном датасете (UnsafeBench).

Что с этим делать — расписано в [PLAN.md](PLAN.md).

> ✅ Закрыто этим рефактором: текст теперь реально модерируется (промпт + OCR);
> zero-shot оценки откалиброваны к safe-якорю; «арбитр» переписан с эвристики на
> прозрачный взвешенный noisy-OR (ушли бессмысленный `evidence_count` и фейковый
> фолбэк); ShieldGemma переосмыслён как эскалационный слой.
